# File Storage Plan

## Managed-Only Portable Storage

Use `app_root/data/database/catalogue.db` for SQLite and `app_root/data/MachineEmbroideryDesigns/` for all embroidery files managed by the app.

Keep the **app-generated preview image** in `Design.image_data` as PNG data, but store the actual embroidery files as normal files under the managed `data/` folder. Simplify to a **single storage model only**: all records are treated as managed, and existing rows can be updated in bulk with a one-off SQL script before release.

---

## Goals

- Make the app portable and suitable for **removable storage** use
- Eliminate dependence on changing drive letters and manually maintained root folders
- Keep implementation complexity low
- Retain the preview image generated by the app in the database
- Store the real embroidery files in a predictable, app-managed folder

---

## Proposed Layout

- **Database:** `app_root/data/database/catalogue.db`
- **Managed files:** `app_root/data/MachineEmbroideryDesigns/`
- **Preview image:** stored in `Design.image_data` as PNG bytes

This means backing up or moving the `data/` folder preserves both:
- the catalogue database
- the imported embroidery files

---

## Plan

### 1. Portable data layout

- Update `src/config.py` so `DATABASE_URL` defaults to:
  - `sqlite:///<app_root>/data/database/catalogue.db`
- Update `DESIGNS_BASE_PATH` to default to:
  - `<app_root>/data/MachineEmbroideryDesigns`
- Ensure startup/bootstrap creates these folders automatically if missing:
  - `data/database/`
  - `data/MachineEmbroideryDesigns/`
- Update `src/services/settings_service.py` help text to match the removable-storage deployment model.

### 2. Minimal schema approach

Keep the schema as simple as possible:

- `Design.filepath` becomes the canonical **managed relative path** under `data/MachineEmbroideryDesigns`
- `Design.image_data` continues to store the **app-created preview PNG** already shown in the UI
- Do **not** store the embroidery file bytes inside the database
- Only add extra columns if they clearly justify the complexity
  - likely candidates: `file_size_bytes`, `sha256`

If simplicity wins, this phase may require **no major schema expansion** beyond standardising how `filepath` is interpreted.

### 3. Import flow becomes always-managed

Update `src/routes/bulk_import.py` and `src/services/bulk_import.py` so imports always:

1. scan a user-selected source folder
2. copy the selected embroidery files into `app_root/data/MachineEmbroideryDesigns/`
3. save the managed relative path into `Design.filepath`
4. keep the preview image in `Design.image_data`

UI simplification:
- no storage-mode choice
- no external-vs-managed toggle
- import messaging should simply explain that files are copied into the catalogue automatically

Update:
- `templates/import/step1_folder.html`
- `templates/import/step2_review.html`

### 4. Open and reveal behavior

Update `_resolve_design_full_path()` in `src/routes/designs.py` so it always resolves the file from:

- the managed base path, plus
- the stored relative `filepath`

From the user’s point of view, these actions remain unchanged:
- `Open in Editor`
- `Show in Explorer`

### 5. UI messaging for a portable managed app

Keep the UI simple and explicit:

- the import screens should state that imported files are copied into the catalogue
- the design detail page should show the managed stored location
- no storage badge is needed if everything is managed

Relevant templates:
- `templates/import/step1_folder.html`
- `templates/import/step2_review.html`
- `templates/designs/detail.html`

### 6. Pre-release data update and documentation

Before release:

- prepare a one-off SQL script to update existing `filepath` values en masse to the new managed-relative convention
- update documentation so users understand that:
  - the app is intended to run from removable storage
  - copying the `data/` folder preserves both metadata and files
  - the original import source folder is no longer required after import

Documentation to update:
- `docs/BACKUP_RESTORE.md`
- `docs/USB_DEPLOYMENT.md`

## Decisions

### Included
- managed-only storage
- portable `data/` layout
- app-generated preview PNG retained in the DB
- pre-release SQL cleanup of existing file paths

### Excluded for now
- external-path compatibility
- DB BLOB storage of embroidery files
- cloud sync
- extra storage-mode UI

### Recommendation
Keep the design simple and let `filepath` always mean:

> relative to `data/MachineEmbroideryDesigns`
